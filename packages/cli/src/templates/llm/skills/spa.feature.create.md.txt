# Make SPA Feature

> **Run autonomously — do not ask the user questions.** When a choice arises, pick the recommended option and proceed.

> **Module location:** a `<module>` usually resolves to `modules/<module>/`, but it can also live under `packages/<module>/` — e.g. once it's been extracted into a shared, publishable package. Check both roots before assuming a path doesn't exist; every `modules/<module>/...` path in this file applies equally under `packages/<module>/...` when that's where the module actually lives.

Generate a SPA feature — a route, its page layout, the route's pending/error/not-found boundaries, and example TanStack Query read/write hooks — then complete the implementation. Follow the shared workflow in the `talos-scaffold` skill (run-from-root, `--name`/`--module` inference, lint/format, and coding conventions); this skill covers only the SPA-feature-specific parts.

## Important

A SPA feature only makes sense inside a **SPA module** (a module whose `type` is `spa` in its `<name>.yml`). Create the SPA first with `spa:create` if it does not exist. The SPA is a standalone Vite front-end: keep features focused on presentation and client state, and reach the backend through HTTP APIs rather than importing server modules. Reuse UI primitives from the linked `design` module instead of duplicating them.

**Place code in the canonical SPA folders — do not invent new ones.** The `### SPA Structure` section of `AGENTS.md` is the source of truth. A feature folder (`features/<feature>/`) and `shared/` share the same fixed sub-layout, and every artefact has exactly one home:

| Sub-folder | What goes there |
|---|---|
| `assets/` | Images/SVG/media used only by this feature |
| `components/` | Presentational + container React components scoped to the feature |
| `hooks/` | Feature React hooks — data fetching, API calls, local UI state |
| `layouts/` | Layout wrappers arranging this feature's pages/sections |
| `services/` | Feature business logic + API clients; the only layer that talks to the backend |
| `store/` | Feature-local client state (signals/store slices), not server data |
| `styles/` | CSS/style modules scoped to this feature |
| `translations/` | i18n dictionary (`translations.json`) + `use<Name>Translate` hook, generated by `talos translation:create` |
| `types/` | TypeScript types/interfaces for the feature's domain |
| `utils/` | Pure helper functions used within the feature |

**Split, don't cram.** As you complete a file, break it into the smallest sensible pieces and move each into its logical folder above rather than leaving everything inside one layout or hook. Reuse the existing folders the generator created and add sibling folders **only** from this fixed list when a piece needs one — never a new name. Anything reused by ≥2 features moves up to `shared/` (same sub-layout); never import another feature's internals.

**One file per element.** Give every component, hook, service, store slice, and util its own `.ts`/`.tsx` file named after it. The **only** exceptions are `types/` and `styles/`: group those **by logic**, not one file per item — e.g. all of the feature's domain types in `types/<kebab>.ts` and its related style rules in `styles/<kebab>.css`, splitting into a second file only when a distinct concern warrants it.

## Steps

### 1. Infer the options from the request, then run the generator

Map the user's request to the options below, then run:

```bash
talos spa:feature:create --name=<name> --module=<module>
```

**Inferring options from the user's request:**

- `--name` — the feature name, taken from the screen or resource it represents (e.g., "a settings page" → `Settings`, "the user profile feature" → `UserProfile`). Pass it in any casing; the CLI normalizes to PascalCase and strips a trailing `Feature` or `Layout` suffix, so omit those suffixes.
- `--module` — the target SPA module, inferred from phrasing like "in the `dashboard` SPA" or "for the admin app". There is **no default** — the generator prompts for it when omitted.
- `--override` — pass when the request is to regenerate an existing feature. Without it, the generator prompts before overwriting and aborts if declined.

Also installs `@tanstack/react-query` if it is not already a dependency.

**Files generated** (under `modules/<module>/src/`, with `<kebab>` the kebab-case feature name and `<Name>` the PascalCase name):

- `routes/<kebab>.tsx` — the TanStack Router file route, wiring the layout and boundaries together
- `features/<kebab>/layouts/<Name>Layout.tsx` — the page layout (the route `component`)
- `features/<kebab>/layouts/<Name>SkeletonLayout.tsx` — the route `pendingComponent`
- `features/<kebab>/layouts/<Name>ErrorLayout.tsx` — the route `errorComponent`
- `features/<kebab>/layouts/<Name>NotFoundLayout.tsx` — the route `notFoundComponent`
- `features/<kebab>/hooks/useGet<Name>.ts` — example TanStack Query read hook + query-key factory
- `features/<kebab>/hooks/useUpdate<Name>.ts` — example TanStack Query mutation hook

### 2. Resolve the linked design module and discover its components

Before writing any UI, find which design module this SPA builds on and what it offers — the layouts must be composed from its components, not styled from scratch.

1. Read the SPA module's yml config at `modules/<module>/<module>.yml` and read the `design:` field. Its value is the kebab-case name of the design module (e.g. `design: "ui"`).
   - If there is **no** `design:` field, the SPA was created without a linked design module. Skip the design-specific guidance below and build the UI with plain elements (suggest the user run `talos spa:create --design <name>` or `talos design:create` if they want a shared design system).
2. List the design module's source to learn the available primitives:

   ```bash
   ls modules/<design>/src/components 2>/dev/null && cat modules/<design>/src/index.ts 2>/dev/null
   ```

   Inspect the component files / barrel exports to see what exists (buttons, cards, inputs, layout primitives, etc.) and their props.
3. Import design components through the module path alias — `@module/<design>/...`:

   ```typescript
   import { Button } from "@module/<design>/components/Button";
   // or, if the design module re-exports from a barrel:
   import { Button, Card } from "@module/<design>";
   ```

   Match the actual export style you found in step 2 (named exports per file vs. a barrel `index.ts`).

### 3. Complete the route

Read `modules/<module>/src/routes/<kebab>.tsx`, then:

- Adjust the route path passed to `createFileRoute` if the feature should not live at `/<kebab>` (e.g. a nested or parameterized path like `/<kebab>/$id`).
- Add a `loader` and `pendingMs` when the page needs data before it renders — the generated `pendingComponent`, `errorComponent`, and `notFoundComponent` exist to cover that loader's pending, failure, and `notFound()` states.
- Keep the boundary components wired to the generated layouts.

### 4. Complete the layouts

Read each file in `modules/<module>/src/features/<kebab>/layouts/` and fill in the real UI, composing the design module's components (resolved in step 2) instead of re-implementing primitives. Keep the layouts as thin arrangers: extract every reusable piece into the canonical folders rather than inlining it. Follow the `optimize-ui` skill's interaction, motion, typography, color, and surface rules throughout — resolve every visual value from the design module's tokens, never a hardcoded one-off. Avoid AI-slop patterns (`optimize-ui`'s `references/ai-slop.md`) — no generic gradient-as-brand-color, stock hero+card-grid skeletons, or emoji standing in for the design system's icons.

- `<Name>Layout` — the page content, arranged from design components and the feature's own components. It renders `children ?? <Outlet />`, so leave the `<Outlet />` in place if the route has children. Pull each presentational block out into `features/<kebab>/components/` (one component per file) and compose them here.
- `<Name>SkeletonLayout` — a skeleton that mirrors the loaded layout's shape; reuse the design's skeleton/placeholder primitive if it has one, and keep the `aria-busy`/`aria-live` attributes for accessibility.
- `<Name>ErrorLayout` — render a useful message from `error` using the design's alert/feedback components, and keep the `reset` retry action (wire it to the design's button).
- `<Name>NotFoundLayout` — the empty/missing-resource state, using the design's empty-state primitives where available.

As you split (one file per element, except types/styles which group by logic):
- presentational/container components → `features/<kebab>/components/` (one component per file)
- feature-local client state → `features/<kebab>/store/` (one slice per file)
- pure helpers (formatting, guards) → `features/<kebab>/utils/` (one helper per file)
- domain types (props/view-model types) → grouped in `features/<kebab>/types/`
- scoped styles → grouped in `features/<kebab>/styles/`
- media → `features/<kebab>/assets/`

If the design module lacks a primitive you need, prefer adding it to the design module (`@module/<design>`) so it stays reusable, rather than styling it inline in the feature.

### 5. Complete the hooks

Read the two hooks in `modules/<module>/src/features/<kebab>/hooks/` and adapt the examples to the real resource, splitting each into its logical folder so the hook stays a thin TanStack Query wrapper:

- Move the resource type (the placeholder `<Name>Type`) out of the hook into the feature's grouped types file (`features/<kebab>/types/<kebab>.ts`) and import it back.
- Move the `fetch`/API-call functions (`get<Name>`, `update<Name>`) into an API client in `features/<kebab>/services/` — services are the only layer that talks to the backend. Keep one service file per element and have the hooks import these instead of calling `fetch` directly.
- `useGet<Name>.ts` — keep only the `useQuery` wiring and the `<camel>Keys` query-key factory; extend the factory with any list/filtered keys the feature needs. The factory is the single source of truth for this feature's keys — reads and invalidations must go through it so they always agree.
- `useUpdate<Name>.ts` — keep only the `useMutation` wiring; keep `onSuccess` seeding the cache via `setQueryData` then returning the `invalidateQueries` promise so the mutation stays pending until the refetch settles.
- Forward the query's `signal` from the hook into the service call to `fetch` so in-flight requests cancel on unmount/refetch.

Add further hooks (lists, additional mutations) in the same folder following these patterns, keeping their fetch logic in `services/` and their types in `types/`.

### 6. Lint, format, and test

```bash
talos monorepo:check
```

Fix every failure before completing.
