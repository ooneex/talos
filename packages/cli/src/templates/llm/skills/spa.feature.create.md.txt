# Make SPA Feature

> **Run autonomously — do not ask the user questions.** When a choice arises, pick the recommended option and proceed.

Generate a SPA feature — a route, its page layout, the route's pending/error/not-found boundaries, and example TanStack Query read/write hooks — then complete it. Follow the shared `talos-scaffold` skill workflow (run-from-root, `--name`/`--module` inference, lint/format, conventions); this covers only the SPA-feature-specific parts.

- **Module location:** `<module>` resolves to `modules/<module>/` or `packages/<module>/` (e.g. once extracted into a shared package). Check both roots before assuming a path is missing.

## Important

A SPA feature only makes sense inside a **SPA module** (`type: spa` in its `<name>.yml`); create it first with `spa:create` if missing. The SPA is a standalone Vite front-end: keep features on presentation and client state, reach the backend through HTTP APIs (never import server modules), and reuse UI primitives from the linked `design` module instead of duplicating them.

**Place code in the canonical SPA folders — never invent new ones.** The `### SPA Structure` section of `AGENTS.md` is the source of truth. A feature folder (`features/<feature>/`) and `shared/` share the same fixed sub-layout; every artefact has exactly one home:

| Sub-folder | What goes there |
|---|---|
| `assets/` | Images/SVG/media used only by this feature |
| `components/` | Presentational + container React components scoped to the feature |
| `hooks/` | Feature React hooks — the only layer that talks to the backend; data fetching, API calls, local UI state |
| `layouts/` | Layout wrappers arranging this feature's pages/sections |
| `services/` | Feature business logic only — pure domain rules; never talks to the backend (hooks do that) |
| `store/` | Feature-local client state (signals/store slices), not server data |
| `styles/` | CSS/style modules scoped to this feature |
| `translations/` | i18n dictionary (`translations.json`) + `use<Name>Translate` hook, generated by `talos translation:create` |
| `types/` | TypeScript types/interfaces for the feature's domain |
| `utils/` | Pure helper functions used within the feature |

**Split, don't cram.** Break each completed file into the smallest sensible pieces, moving each into its logical folder above rather than leaving everything in one layout or hook. Reuse the generator's folders; add sibling folders **only** from this fixed list — never a new name. Anything reused by ≥2 features moves up to `shared/` (same sub-layout); never import another feature's internals.

**One file per element.** Every component, hook, service, store slice, and util gets its own `.ts`/`.tsx` file named after it. **Only** exceptions: `types/` and `styles/` group **by logic** — domain types in `types/<kebab>.ts`, style rules in `styles/<kebab>.css` — splitting into a second file only when a distinct concern warrants it.

## Steps

### 1. Infer the options from the request, then run the generator

```bash
talos spa:feature:create --name=<name> --module=<module>
```

- `--name` — feature name, from the screen or resource it represents ("a settings page" → `Settings`, "the user profile feature" → `UserProfile`). Any casing; the CLI normalizes to PascalCase and strips a trailing `Feature`/`Layout` suffix, so omit those.
- `--module` — target SPA module, inferred from phrasing like "in the `dashboard` SPA" or "for the admin app". **No default** — prompts when omitted.
- `--override` — pass to regenerate an existing feature. Without it, the generator prompts before overwriting and aborts if declined.

Also installs `@tanstack/react-query` if not already a dependency.

**Files generated** (under `modules/<module>/src/`, `<kebab>` kebab-case feature name, `<Name>` PascalCase):

- `routes/<kebab>.tsx` — the TanStack Router file route, wiring the layout and boundaries together
- `features/<kebab>/layouts/<Name>Layout.tsx` — the page layout (the route `component`)
- `features/<kebab>/layouts/<Name>SkeletonLayout.tsx` — the route `pendingComponent`
- `features/<kebab>/layouts/<Name>ErrorLayout.tsx` — the route `errorComponent`
- `features/<kebab>/layouts/<Name>NotFoundLayout.tsx` — the route `notFoundComponent`
- `features/<kebab>/hooks/useGet<Name>.ts` — example TanStack Query read hook + query-key factory
- `features/<kebab>/hooks/useUpdate<Name>.ts` — example TanStack Query mutation hook

### 2. Resolve the linked design module and discover its components

Before writing UI, find which design module this SPA builds on — layouts must be composed from its components, not styled from scratch.

1. Read `modules/<module>/<module>.yml`'s `design:` field — a kebab-case design-module name (e.g. `design: "ui"`).
   - **No `design:` field** — no linked design module. Skip the design guidance below, build with plain elements, and suggest `talos spa:create --design <name>` or `talos design:create` for a shared design system.
2. List the design module's source to learn its primitives (buttons, cards, inputs, layout primitives, etc.) and their props:

   ```bash
   ls modules/<design>/src/components 2>/dev/null && cat modules/<design>/src/index.ts 2>/dev/null
   ```
3. Import via the `@module/<design>/...` alias, matching the export style you found (per-file vs. barrel `index.ts`):

   ```typescript
   import { Button } from "@module/<design>/components/Button";
   // or, if the design module re-exports from a barrel:
   import { Button, Card } from "@module/<design>";
   ```

### 3. Complete the route

Read `modules/<module>/src/routes/<kebab>.tsx`, then:

- Adjust the `createFileRoute` path if the feature should not live at `/<kebab>` (e.g. nested/parameterized like `/<kebab>/$id`).
- Add a `loader` and `pendingMs` when the page needs data before rendering — the generated `pendingComponent`, `errorComponent`, and `notFoundComponent` cover that loader's pending, failure, and `notFound()` states.
- Keep the boundary components wired to the generated layouts.

### 4. Complete the layouts

Read each file in `modules/<module>/src/features/<kebab>/layouts/` and fill in the real UI, composing the design module's components (step 2) instead of re-implementing primitives. Keep layouts as thin arrangers: extract every reusable piece into the canonical folders. Follow the `optimize-ui` skill's interaction, motion, typography, color, and surface rules — resolve every visual value from the design module's tokens, never a hardcoded one-off. Avoid AI-slop (`optimize-ui`'s `references/ai-slop.md`) — no generic gradient-as-brand-color, stock hero+card-grid skeletons, or emoji for the design system's icons.

- `<Name>Layout` — page content arranged from design and feature components. Renders `children ?? <Outlet />`, so leave `<Outlet />` in place if the route has children. Pull each presentational block into `features/<kebab>/components/` (one per file) and compose here.
- `<Name>SkeletonLayout` — a skeleton mirroring the loaded layout's shape; reuse the design's skeleton/placeholder primitive if any, and keep the `aria-busy`/`aria-live` attributes.
- `<Name>ErrorLayout` — render a useful message from `error` via the design's alert/feedback components; keep the `reset` retry action (wire to the design's button).
- `<Name>NotFoundLayout` — the empty/missing-resource state, using the design's empty-state primitives where available.

As you split (one file per element, except types/styles which group by logic):
- presentational/container components → `features/<kebab>/components/` (one component per file)
- feature-local client state → `features/<kebab>/store/` (one slice per file)
- pure helpers (formatting, guards) → `features/<kebab>/utils/` (one helper per file)
- domain types (props/view-model types) → grouped in `features/<kebab>/types/`
- scoped styles → grouped in `features/<kebab>/styles/`
- media → `features/<kebab>/assets/`

If the design module lacks a primitive you need, add it to the design module (`@module/<design>`) so it stays reusable, rather than styling inline in the feature.

### 5. Complete the hooks

Read the two hooks in `modules/<module>/src/features/<kebab>/hooks/` and adapt the examples to the real resource. **Hooks are the only layer that talks to the backend** — keep `fetch`/API calls here, and push everything else into its logical folder so the hook stays a focused TanStack Query wrapper:

- Move the resource type (placeholder `<Name>Type`) out of the hook into the grouped types file (`features/<kebab>/types/<kebab>.ts`) and import it back.
- Keep `fetch`/API calls (`get<Name>`, `update<Name>`) in `hooks/` — never move backend access into `services/`. Extract only the **pure domain logic** the hook needs — request/response mapping, validation, business rules — into `features/<kebab>/services/` (one file per element); services never call the backend.
- `useGet<Name>.ts` — keep the `useQuery` wiring, its backend `fetch`, and the `<camel>Keys` query-key factory; extend the factory with any list/filtered keys needed. The factory is the single source of truth for this feature's keys — reads and invalidations must go through it so they always agree.
- `useUpdate<Name>.ts` — keep the `useMutation` wiring and its backend call; keep `onSuccess` seeding the cache via `setQueryData` then returning the `invalidateQueries` promise so the mutation stays pending until the refetch settles.
- Forward the query's `signal` into the hook's `fetch` so in-flight requests cancel on unmount/refetch.

Add further hooks (lists, mutations) in the same folder following these patterns — backend calls in the hook layer, pure domain logic in `services/`, types in `types/`.

### 6. Write a test for every element

**No element ships untested.** As you split code into the canonical folders, add a matching spec mirroring `src/` under `modules/<module>/tests/`, using `bun:test` and the `optimize-testing` conventions (meaningful behavior only — no trivial getters or placeholder assertions). Prefer the `talos react:component:create` generator for feature components — it scaffolds a happy-dom + React Testing Library spec you then expand.

| Element | Test location | What to cover |
|---|---|---|
| Layout / component (`.tsx`) | `tests/features/<kebab>/layouts/<Name>.spec.tsx` (or `.../components/`) | renders, each state (loaded / skeleton / error / not-found), meaningful props, user interactions |
| Hook (`useGet<Name>`, `useUpdate<Name>`) | `tests/features/<kebab>/hooks/<hook>.spec.ts` | loading / success / error, the query-key factory, cache seeding + invalidation (mock the backend `fetch`/client — the hook is the backend layer) |
| Service | `tests/features/<kebab>/services/<name>.spec.ts` | pure domain logic — request/response mapping, validation, business rules (no network) |
| Store slice | `tests/features/<kebab>/store/<name>.spec.ts` | state transitions and selectors |
| Util | `tests/features/<kebab>/utils/<name>.spec.ts` | behavior + edge cases (empty / boundary inputs) |
| Translation | `tests/features/<kebab>/translations/<name>.spec.ts` | each locale resolves the expected keys; `use<Name>Translate` returns them (the `talos translation:create` generator scaffolds this spec) |

Query components by role/text/label (not test IDs) and assert with jest-dom matchers.

### 7. Lint, format, and test

```bash
talos monorepo:check
```

Fix every failure before completing.
